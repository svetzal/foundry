# Maintenance Takeover Audit

**Goal:** ensure foundry has fully taken over nightly maintenance from the legacy `maintain.sh` system, with no gaps in visibility or functionality.

**Status:** in progress — legacy launchd job disabled 2026-04-14; verification ongoing.

**Context:** Until 2026-04-14, two independent maintenance systems ran nightly at 2:00 AM:

- **Legacy:** `com.mojility.maintenance` launchd job → `Automation/maintenance/maintain.sh` → `hone` CLI, with its own registry at `Automation/maintenance/registry.json`.
- **Current:** `com.mojility.foundry-maintenance` launchd job → `foundry run` → foundryd, with registry at `~/.foundry/registry.json`.

Both wrote to `Automation/maintenance-audits/runs/YYYY-MM-DD/summary.md`, clobbering each other based on which finished last. On 2026-04-14, legacy was disabled via `launchctl bootout` and the plist renamed to `.disabled`.

## Design Principle

Events in `~/.foundry/events/` and traces in `~/.foundry/traces/` are the detailed source of truth for all maintenance activity. Markdown summaries (per-project action narratives, nightly run summaries) are **derived views**, not primary records. This lets us:

- Render the same events into multiple formats (terse summary, full narrative, dashboard, Slack post) without data duplication.
- Fix a rendering bug by re-running the summarizer against historical events rather than losing history.
- Add new derived views (e.g. weekly trend reports) as needed without touching the collection layer.

Per-project markdown writes should be implemented as a terminal block triggered by run-completion events, not as side effects of the execution blocks.
